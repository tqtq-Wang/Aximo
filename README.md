# Axiom

## 中文

### 项目简介

Axiom 是一个面向人类与 AI 协作开发的静态强类型编译语言项目。

这个仓库是 Axiom 语言项目的 monorepo 初始化版本，当前优先建立稳定目录结构、显式语义约束、机器可读编译输出契约，以及适合多 Codex 并行协作的开发底盘，而不是抢跑复杂编译器实现。

当前仓库的直接目标是提供：

- 稳定的文档与规范表面
- 统一术语基线
- AST、diagnostics、compiler summary 的 JSON Schema 初稿
- 可作为 parser 与 tooling 夹具的示例项目
- 多 AI 并行协作规则
- 面向真实后端工程演进的 roadmap

### 为什么是 Axiom

Axiom 的核心假设是：未来代码会长期被人类和 AI 一起阅读、修改、验证和局部生成。因此，语言层就要把下面这些能力视为一级要素：

- 显式可见性
- 显式错误传播
- 显式 effects
- 固定项目与模块结构
- 结构化 diagnostics
- 结构化 compiler summary 输出

这个仓库还承载一个更务实的目标：Axiom 不是语言玩具，而是计划逐步进入 AI 辅助的后端服务开发，替代部分现有后端语言在结构稳定性、语义可提取性和自动化协作上的不足。

### 仓库结构

- `docs/`: VitePress 文档站与项目规范事实源
- `schemas/`: AST、diagnostics、compiler summary 的 JSON Schema
- `examples/`: 示例 Axiom 项目，同时作为文档和未来编译器夹具
- `tests/`: parser、diagnostics、snapshot 夹具布局
- `compiler/`: 未来编译器组件的实现工作区
- `tools/scripts/`: 校验脚本与 snapshot 生成脚本
- `.github/workflows/`: 最小 CI，用于文档和仓库契约检查

### 事实源优先级

当多个文件内容存在重叠时，遵循以下优先级：

1. `docs/spec/*.md`
2. `docs/decisions/*.md`
3. `docs/rfc/*.md`
4. `schemas/*.json`
5. `examples/*`

本仓库初始化所依据的原始设计文档是 [`Axiom Language RFC v0.1.md`](./Axiom%20Language%20RFC%20v0.1.md)。仓库中的文档是在此基础上做工程化整理和规范化落地。

### 核心原则

- 显式优于隐式
- 稳定结构优于局部便利
- 机器可读语义是产品的一部分
- examples 必须持续与 spec 保持一致
- 超出 RFC v0.1 的变更必须显式更新文档

## English

### Overview

Axiom is a statically typed compiled language project for human and AI collaborative development.

This repository is the monorepo bootstrap for the language itself. At this stage, it prioritizes stable structure, explicit semantic contracts, machine-readable compiler outputs, and a workflow suitable for multiple Codex agents working in parallel, instead of rushing into a complex compiler implementation.

The immediate goals of this repository are to provide:

- a stable documentation and specification surface
- a shared terminology baseline
- initial JSON Schema drafts for AST, diagnostics, and compiler summary
- example projects that also serve as parser and tooling fixtures
- parallel AI collaboration rules
- a roadmap toward practical backend-oriented usage

### Why Axiom

Axiom is based on a simple assumption: code will increasingly be read, modified, validated, and partially generated by both humans and AI systems. Because of that, the language treats the following as first-class concerns:

- explicit visibility
- explicit error propagation
- explicit effects
- fixed project and module layout
- structured diagnostics
- structured compiler summary output

This repository also has a practical product direction. Axiom is not intended to remain a toy language. It is being shaped toward AI-assisted backend service development, with the goal of gradually replacing parts of existing backend stacks where stronger structure, analyzable semantics, and automation-friendly workflows matter.

### Repository Layout

- `docs/`: VitePress documentation site and source of truth for repository-facing specs
- `schemas/`: JSON Schema drafts for AST, diagnostics, and compiler summary
- `examples/`: example Axiom projects used as documentation and future compiler fixtures
- `tests/`: parser, diagnostics, and snapshot fixture layout
- `compiler/`: implementation work areas for future compiler components
- `tools/scripts/`: validation and snapshot generation scripts
- `.github/workflows/`: minimal CI for docs and repository contract checks

### Source-of-Truth Order

When files overlap in meaning, use the following precedence:

1. `docs/spec/*.md`
2. `docs/decisions/*.md`
3. `docs/rfc/*.md`
4. `schemas/*.json`
5. `examples/*`

The original design input for this bootstrap is [`Axiom Language RFC v0.1.md`](./Axiom%20Language%20RFC%20v0.1.md). The repository documents normalize that RFC into an execution-ready project baseline.

### Current Status

This is still a repository bootstrap milestone, not a completed compiler milestone.

Included today:

- document architecture
- language specification drafts
- RFCs and decision records
- schema drafts
- example projects
- test fixture conventions
- minimal CI and validation scripts
- multi-agent task slicing documents

Explicitly not included today:

- full lexer
- full parser
- type checker
- effect checker
- runtime or standard library implementation

### Principles

- explicit beats implicit
- stable structure beats local convenience
- machine-readable semantics are part of the product
- examples must remain aligned with the spec
- changes beyond RFC v0.1 require explicit documentation updates
